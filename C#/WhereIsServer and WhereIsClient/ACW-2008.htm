Client and Server in C#

The coursework requires you to implement two C# Windows console applications; one is a client and the other is the server. They should use TCP sockets by means of the appropriate classes in System.Net and System.Net.Sockets. The task is to implement a simple student locating facility. Students will be identified by their computer login name. It is not necessary to implement any form of user interface and simple use of Microsoft Command Prompt interface to run the client is required. If students implement an optional windows interface this should be enabled by a command line switch.

The client, we shall call it whereis can have two arguments, a user name and a string giving location information.

For example: 

G:\08241\> whereis cssbct
cssbct is in RB-336
G:\08241\> whereis cssbct "in RB-310 for a meeting"
cssbct location changed to be in RB-310 for a meeting
G:\08241\> whereis cssbct
cssbct is in RB-310 for a meeting

In this example the first call of the client showed the current location of the user cssbct. The second was used to change the location and the third showed the changed response.

Protocol for Coursework

Introduction

Your coursework should implement the following protocol for communication between the client and server. If each student invented their own protocol, the there would be no interoperability between different implementations. Students would also be making assumptions about requirements when implementing their programs which is not generally a good action for a software engineer. Requiring the implementation of a standard protocol means that every student's implementation should be interoperable with every other, and the working server provided by the lecturer. So you can develop your client using the lecturer's server, and develop your server later once you are happy with the correctness of your client. It is necessary to communicate with a reference or external server in addition to your own server in order to demonstrate sucessful client operation for assessment.

You will encode your data to be transferred between client and server as US-ASCII strings terminated by US-ASCII characters 13 then 10 (carriage return then line feed). There are other more readily available data encoding options in C# such as unicode and object serialization, but using US-ASCII makes our communication easily understandable by other non-C# applications.

Protocol

  1. The server waits for clients to contact it.
  2. Clients should connect and communicate with the server via sockets and TCP packets.
  3. Our protocol will operate over port 43, normally used by the internet whois protocol.
  4. A request from the client to the server for the server to query the database for a persons location looks like this:
    <name><CR><LF>
    Where <name> is the name of the person who's location the client is requesting.
  5. A request from the client to the server for the server to update the database with a new location for a person looks like this:
    <name><space><location><CR><LF>
    Where: <name> is the name of the person whos location the client wants to update.
    <space> is the US-ASCII space character.
    <location> is the new location for that person.
  6. When the client requests the server to query the database for a persons location and the server finds an entry in the database for that person the server responds like this:
    <location><CR><LF>
    <drop connection>
    Where: <location> is the location if found in the database for the person.
  7. When the client requests the server to query the database for a persons location but the server is unable to find an entry in the database for that person the server responds like this:
    ERROR: no entries found<CR><LF>
    <drops connection>
  8. When the client sends a request to the server to update the database with a new location for a person the server responds like this:
    OK<CR><LF>
    <drops connection>

In all examples above <CR> is US-ASCII character number 13, the carriage return character, and <LF> is US-ASCII character number 10, the line feed character. We are using both together as our string terminator. In all cases both the client and server should give up and drop connection if they are left waiting more than 1 second (1000 milliseconds).

Optional Features

The client, as a minimum, can have the IP address of the server hardwired into the C# code. This is not very useful, and would be better to allow a parameter to specify the server IP. Even better, would be a Fully Qualified Domain Name for the server. For example: 

G:\08241\> whereis -h 150.237.92.99 cssbct
cssbct is in RB-321
G:\08241\> whereis -h somepc.dcs.hull.ac.uk cssbct "changed room"
cssbct location changed to changed room

The server could, at a minimum, do its job silently. It would be helpful if it reported activity, either to the command window, or to a specially designed administration interface. For example: 

G:\08241\> whereisserver
request for cssbct
replied in RB-321

The more information that is recorded the better, and a good server should keep log files: 

200.4.239.13 - - [28/Dec/2003:03:14:56 +0000] "GET cssbct" OK
150.237.92.99 - - [28/Dec/2003:03:15:23 +0000] "SET cssbct some room" OK
150.237.92.99 - - [28/Dec/2003:03:19:24 +0000] "GET cshwvv" UNKNOWN

This example is shown in "Standard Log Format".

Deliverables

You must submit the following:

  1. A report on your work, which should include, as an appendix, the listings of your client and server programs, clearly marked and appropriately annotated with comments. You should report on how you have tested the work and show command prompt samples of input and output from the testing which demonstrate to the reader its operation without the need to run programs from the disk. The input output samples could be by including screen captures in your report or redirecting the command prompt input output to a file which is included in the report. You should comment on aspects of your design.
  2. An electronic copy of your source code, compiled class files and report on either floppy disk or CD, bundled securely with the report. The disks should be labeled with your student number and module number and year (2008) so they can be identified if they become seperated from the report! The disk should also contain a file named readme.txt which contains the same information as written on the disk label and an annotated inventory of disk contents. If submitting a CD a flexible plastic or paper case is preferable to a jewel case which is too bulky! The programs may be run from the disk to clarify aspects of the operation of your work or selected at random for checking purposes, and thus must be runnable from CD without installation.
  3. You may be asked to demonstrate the operation of your programs in the laboratory as part of the assessment. If you have developed the code elsewhere, please ensure it is compatible and available to be demonstrated on university lab systems.

Suggested Timescale for Completion of Work

Weeks 1 - 3. Develop and compete your client by having it communicate with the working server provided by your lecturer or an external third party internet server. Ask your  demonstrator or lecturer for the IP address or domain name of the server. It is often sensible to be sure of the correctness of your client before proceeding to develop your server.

Weeks 4 - 6. Develop and complete your server using your own client to interface with it. Note that to truly test the network communication between your client and server both programs need to be running on separate machines.

Weeks 6 - 7. Complete the documentation and report.
